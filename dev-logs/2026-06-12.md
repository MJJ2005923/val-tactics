## 踩坑记录

### 1. sessionStorage vs localStorage 不一致
- **现象**：协作频道静默失效
- **原因**：App.tsx 写 sessionStorage，TacticsContext 读 localStorage
- **教训**：所有跨组件通信的存储键统一用 localStorage

### 2. Supabase Realtime WebSocket 回退 REST
- **现象**：广播延迟高、不可靠
- **原因**：公网环境 WebSocket 连接不稳定，自动 fallback 到 HTTP polling
- **教训**：实时广播需要稳定的 WebSocket，国内访问 Supabase 延迟不可控

### 3. RLS 鸡生蛋问题
- **现象**：创建房间后无法读取（"数据库未响应"）
- **原因**：rooms_read 策略要求是成员才能读，但创建室时还未加入成员表
- **教训**：RLS 设计与业务流程需要仔细对齐

### 4. PostgREST headers 大小写
- **现象**：房间 status 默认为 closed 而非 open
- **原因**：SB_HEADERS 用了大写 `Content-Type`，PostgREST 可能不认
- **教训**：HTTP headers 虽说是大小写不敏感，但具体实现有差异

### 5. myUserId 随机字符串 ≠ UUID
- **现象**：编辑权锁定从未生效
- **原因**：myUserId 是 `'u' + Math.random()` 随机串，roomEditorId 是 Supabase UUID
- **教训**：需要匹配的 ID 必须从同一个数据源获取

### 6. canCreateRoom 过期判断颠倒
- **现象**：刚激活就显示"套餐已过期"
- **原因**：`激活时间 < 当前时间` 而非 `激活时间 + 30天 < 当前时间`
- **教训**：时间戳语义要明确——是激活点还是过期点

### 7. 空快照覆盖画布
- **现象**：观察者加入后画面空白
- **原因**：canvas_snapshot 默认值 `{}` 被视为有效快照，覆盖了 Realtime 传来的初始状态
- **教训**：轮询方案需要判空机制和延迟启动

### 8. Realtime 广播不可靠（根本问题）
- **现象**：UPDATE_ABILITY_SHAPE 不送达观察者
- **原因**：WebSocket 不稳定 + REST fallback 只支持发送不支持订阅接收
- **教训**：Supabase Realtime 不适合需要可靠同步的协作场景

## 核心教训总结

> **多人实时协作超出了当前技术栈的能力边界。**
>
> Supabase Realtime 在公网（国内→新加坡）环境下 WebSocket 不稳定，REST fallback 单向不可靠，RLS 与业务流程耦合复杂。真正的协作需要专用技术栈（WebRTC 数据通道 / 第三方 RT 服务），不适合在当前项目强行实现。
>
> **及时止损比死磕更重要。** 本次在 48 小时内尝试了 30+ 次提交、6 种方案（增量广播 → Realtime 快照 → API 迁移 → 轮询快照），最终确认不可行后果断移除。
